# Active vs Passive Voice in PR Descriptions

**Active voice** (“I added validation”) is direct; **passive** (“validation was added”) can sound evasive. Technical PRs benefit from **clarity** and **accountability**, not stiff corporate passive.

---

## Good PR title (active)

- **“Add idempotency key header to checkout API”** — verb-led, scoped.

## Weak title (vague passive)

- **“Changes were made”** — no searchable signal.

---

## Body structure

1. **What** changed (bullet list).
2. **Why** (problem + link to ticket).
3. **How to test** (commands, screenshots for UI).
4. **Risk / rollout** (feature flag, migration).

---

## When passive is OK

Generated notes (“**lint was auto-fixed**”)—still include **who** can revert if needed.
